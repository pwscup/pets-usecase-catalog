# プロンプト: 文献から事例JSONを作成する

以下のプロンプトをAI（Claude等）に入力し、文献のURLやPDFを添付することで、`case.json` のドラフトを生成できます。

---

## 使い方

1. 下記のプロンプトをコピーする
2. `【ここに文献のURL、PDF本文、またはWebページのテキストを貼り付けてください。】` の部分を実際の情報に差し替える
3. AIに送信する
4. 出力されたJSONを事例追加フォームの「AIの出力をインポート」に貼り付ける

> **Tip**: システムの事例追加画面にある「AI で入力を補助する」パネルから、最新のプロンプトをコピーできます。

---

## プロンプト本文

````
# プライバシー強化技術（PETs）活用事例の作成

## 入力情報（※ ここに参考文献を追記してください）

以下の【 】内に、参考にする文献の情報を貼り付けてから、このプロンプトをAIに送信してください。
URL、PDFのテキスト、Webページの本文など、どの形式でも構いません。

【
ここに文献のURL、PDF本文、またはWebページのテキストを貼り付けてください。
例: https://example.com/privacy-tech-report.pdf
例: 論文タイトル「○○におけるプライバシー強化技術の活用」の本文テキスト
】

---

上記の情報から、プライバシー強化技術（PETs: Privacy Enhancing Technologies）の活用事例を抽出し、case.json を作成してください。

対象となる技術カテゴリ:
- 合成データ（Synthetic Data）
- 差分プライバシー（Differential Privacy）
- 匿名化（Anonymization / k-anonymity / l-diversity 等）
- 連合学習（Federated Learning）
- 秘密計算（Secure Computation / MPC / TEE 等）
- 分散データ分析（Distributed Analytics）

## 出力仕様

1件の事例につき1つのJSONを出力してください。文献中に複数の事例が含まれる場合は、事例ごとに分けて出力してください。

各JSONは以下のスキーマに従ってください：

```json
{
  "title": "（後述のガイドラインに従って作成）",
  "region": "国内, 国外 のいずれか",
  "domain": "金融, 医療, 公共, 通信, モビリティ, IT, エネルギー, 小売, 製造 のいずれか",
  "domain_sub": "分野のサブカテゴリ（任意。例: 保険, 製薬, がん臨床）",
  "organization": "実施組織名",
  "technology_category": ["下記の技術カテゴリから1つ以上"],
  "usecase_category": ["下記のユースケース分類から1つ以上"],
  "occurred_at": "事例の発生時期（YYYY, YYYY-MM, または YYYY-MM-DD。不明の場合は null）",
  "summary": "（後述のガイドラインに従って作成）",
  "value_proposition": "成果・効果の具体的な説明",
  "privacy_enhancement_method": "プライバシー強化手法の説明（使用技術、パラメータ等）",
  "safety_assurance_method": "安全性保証手法（再識別リスク評価、監査、アクセス制御 等）",
  "utility_evaluation_method": "有用性の評価方法（精度比較, F値, 下流タスク評価 等）",
  "tags": ["検索用のキーワード2〜4個"],
  "sources": [
    {
      "source_type": "web or pdf",
      "title": "文献の正式タイトル",
      "url": "文献のURL",
      "note": "この文献からどの情報を参照したかの補足"
    }
  ],
  "figures": [
    {
      "type": "data_flow",
      "title": "（事例名）のPETs活用フロー",
      "data": {
        "nodes": [
          {"id": "s1", "label": "元データの説明",         "category": "source"},
          {"id": "s2", "label": "課題・制約の説明",        "category": "constraint"},
          {"id": "p1", "label": "プライバシー強化手法",     "category": "process"},
          {"id": "a1", "label": "データ・モデルの活用先",   "category": "application"},
          {"id": "a2", "label": "得られた成果・効果",       "category": "outcome"}
        ],
        "edges": []
      },
      "note": ""
    }
  ]
}
```

## technology_category の選択肢

文献で使用されている技術から最も適切なものを選んでください（複数選択可）：

- synthetic_data（合成データ）
- differential_privacy（差分プライバシー）
- anonymization（匿名化）
- federated_learning（連合学習）
- secure_computation（秘密計算）
- distributed_analytics（分散データ分析）

| 技術カテゴリ | 判断基準 |
|-------------|---------|
| synthetic_data | 元データの統計的特性を保持した合成データを生成して活用する |
| differential_privacy | ノイズ付加等により差分プライバシー保証を提供する |
| anonymization | k-匿名性、l-多様性、t-近接性等の匿名化手法を適用する |
| federated_learning | データを集約せず分散環境でモデルを学習する |
| secure_computation | 秘密分散、準同型暗号、TEE等で暗号化したままデータを処理する |
| distributed_analytics | データを移動させず各拠点で分析し結果のみを集約する |

## usecase_category の選択肢

文献の内容から最も適切なものを選んでください：

- 組織内データ共有
- 組織間データ共有
- 外部分析者活用
- R&D
- データ販売
- フィージビリティ検証
- 公的利用

| カテゴリ | 判断基準 |
|---------|---------|
| 組織内データ共有 | 同一組織内の部門間でデータを共有するためにPETsを使う |
| 組織間データ共有 | 異なる組織（取引先、パートナー等）にデータを提供するためにPETsを使う |
| 外部分析者活用 | 外部の分析者・研究者にデータを提供して分析してもらう |
| R&D | 研究開発・学術研究の目的でPETsを活用する |
| データ販売 | プライバシーを保護したデータセットを商品として販売する |
| フィージビリティ検証 | 本格導入前の実現可能性検証・PoC用途でPETsを使う |
| 公的利用 | 行政・統計機関等が公的目的でPETsを活用する |

## domain の選択肢

金融, 医療, 公共, 通信, モビリティ, IT, エネルギー, 小売, 製造

## region の選択肢

国内, 国外

## 記入ガイドライン

### title
- 「〇〇における△△を目的としたPETs活用」のように、対象領域と目的が分かる形にする
- 使用技術が明確な場合は技術名を含める（例:「連合学習による多施設共同研究」「差分プライバシーを適用した統計公開」）
- 具体的な成果があれば盛り込む（例:「承認プロセスを6カ月から3日に短縮」）
- 30〜50文字程度を目安にする

### summary
- 「課題 → PETsの活用 → 結果」の構造で1〜3文にまとめる
- 1文目で課題や背景、2文目でどのPETsをどう使ったか、3文目で成果を書く

### occurred_at
- 文献の公開日や事例の実施時期から、事例の発生時期を記入する
- 年のみ判明: "2023"、月まで判明: "2023-06"、日まで判明: "2023-06-15"
- 時期が不明な場合は null とする

### data_flow 図のノード
- 各カテゴリに1〜3ノードが目安
- label は短く具体的に（15〜25文字程度）
- 数値があれば盛り込む（例:「有用性97%を達成」「6カ月→3日に短縮」）
- ノードIDは s1, s2（source列）、p1, p2（process列）、a1, a2（application/outcome列）の命名規則

### 情報が不明な場合
- 文献から読み取れない項目は「調査中」と記入する（空欄にしない）
- 無理に推測で埋めず、正確に分かる範囲だけを記入する

### tags
- 検索で引っかかるべきキーワードを2〜4個
- domain や usecase_category と重複しないものを選ぶ

## 出力形式

JSONのみを出力してください（マークダウンのコードブロックで囲む）。
複数事例の場合は、事例ごとに別のコードブロックで出力し、間に簡単な説明を挟んでください。
````

## 補足: 複数文献から1つの事例を作る場合

複数の文献を同時に渡して「これらの情報を統合して1つの事例にまとめてください」と追記すれば、情報を統合した事例を作成できます。その場合、`sources` 配列に全ての文献を列挙してください。
